# Logs 101: `.tlog` and `.bin`

ArduPilot gives us two very useful log families:

- `.tlog`: telemetry log recorded by a ground station.
- `.bin`: onboard DataFlash log recorded by the flight controller.

They can describe the same flight from different viewpoints.

After this page, read [ArduPilot Log Field Guide](05-ardupilot-log-field-guide.md) for a
practical guide to terms seen in `report.md`, `map.html`, `events.json`, and `track.csv`.

## Quick Comparison

| Topic | `.tlog` | `.bin` |
| --- | --- | --- |
| Full name | Telemetry log | ArduPilot DataFlash log |
| Recorded by | Ground station or MAVLink receiver | Flight controller onboard storage |
| Main format | MAVLink packet stream | ArduPilot DataFlash records |
| Best evidence for | What the operator/GCS received | What the autopilot recorded onboard |
| Common weakness | Telemetry dropouts, late connection, low message rate | Requires aircraft/storage access, logging may be disabled |
| Parser in this repo | `pymavlink.mavlink_connection()` | `pymavlink` DataFlash reader |

## What Is a `.tlog`?

A `.tlog` records MAVLink telemetry messages sent between the autopilot and the ground station. Mission Planner creates telemetry logs when connected to the vehicle, and QGroundControl can also work with MAVLink telemetry logs.

Think of a `.tlog` as the ground-side view of the flight. It can be excellent for
understanding what the ground station received, but it may miss data if the telemetry
radio/link dropped packets, connected late, or did not request a high-rate message stream.

Useful MAVLink messages:

| Message | Why It Matters |
| --- | --- |
| `HEARTBEAT` | Identifies vehicle type, autopilot type, system state |
| `AUTOPILOT_VERSION` | Firmware version, board/version fields, capabilities, UID fields when sent |
| `SYSTEM_TIME` | Helps connect boot time to UTC |
| `GLOBAL_POSITION_INT` | Main estimated latitude, longitude, altitude, velocity, heading |
| `GPS_RAW_INT` | Raw GPS fix, satellites, GPS altitude, accuracy hints |
| `VFR_HUD` | User-facing speed, altitude, heading, climb |
| `HOME_POSITION` | Home/launch point |
| `ATTITUDE` | Roll, pitch, yaw |
| `BATTERY_STATUS` | Battery state |
| `SYS_STATUS` | System health and battery summary |
| `RC_CHANNELS` | Pilot RC input channels |
| `SERVO_OUTPUT_RAW` | Motor/servo output commands |
| `STATUSTEXT` | Human-readable autopilot warnings/messages |
| `PARAM_VALUE` | Parameter values sent to the ground station |
| `COMMAND_LONG` / `COMMAND_ACK` | Commands and acknowledgements seen over MAVLink |

Example from this repo:

```bash
.venv/bin/python scripts/read_log.py logs/ArduCopter-test.tlog --limit 30 -o logs/ArduCopter-test-preview.txt
```

Expected kind of summary:

```text
Vehicle: MAV_TYPE_QUADROTOR
Autopilot: MAV_AUTOPILOT_ARDUPILOTMEGA
Vehicle source: sys=1 comp=1
GCS source: sys=255 comp=230
```

## What Is a `.bin`?

A `.bin` is an ArduPilot DataFlash log. It is usually stored by the flight controller itself. It is not the same as a DJI `.DAT` file.

Think of a `.bin` as the onboard view of the flight. It is usually stronger for flight
reconstruction because it is written by the flight controller, often with more record types
and higher-rate sensor data than a telemetry log. Its weakness is practical: you need access
to the aircraft, SD card, or onboard log download, and logging must have been enabled.

Useful DataFlash record types:

| Record | Why It Matters |
| --- | --- |
| `GPS` | GPS fix, satellites, lat/lon, altitude, speed |
| `POS` | Estimated position and relative altitude |
| `BARO` | Barometer altitude, pressure, temperature, climb rate |
| `ATT` | Roll, pitch, yaw |
| `MODE` | Flight mode changes |
| `EV` | Events such as arming/disarming/failsafe-related markers |
| `ERR` | Subsystem errors |
| `BAT` | Battery voltage, current, consumed energy |
| `IMU` | Accelerometer and gyroscope data |
| `MAG` | Compass/magnetometer data |
| `RCIN` | Pilot RC input values |
| `RCOU` | Motor/servo output values |
| `PARM` | Parameters logged with the flight |
| `MSG` | Autopilot messages |
| `VER` | Firmware and board version where present |

Example from this repo:

```bash
.venv/bin/python scripts/read_log.py logs/sample-arducopter.bin --limit 30 -o logs/sample-arducopter-preview.txt
```

## How the Same Flight Can Produce Both

```mermaid
sequenceDiagram
    participant Drone as ArduPilot Drone
    participant FC as Flight Controller Storage
    participant Link as Telemetry Link
    participant GCS as Ground Station

    Drone->>FC: Write onboard DataFlash records
    Drone->>Link: Send MAVLink telemetry
    Link->>GCS: Deliver telemetry packets
    GCS->>GCS: Record .tlog
    FC->>FC: Store .bin
```

For best evidence, collect both:

- `.bin` says what the autopilot recorded onboard.
- `.tlog` says what the ground side saw or commanded.

## Important Units

MAVLink often uses scaled integers:

```text
lat/lon in GLOBAL_POSITION_INT: degrees * 1e7
alt in GLOBAL_POSITION_INT: millimeters above mean sea level
relative_alt: millimeters relative to home
vx/vy/vz: centimeters per second
hdg: centidegrees
```

ArduPilot DataFlash logs often expose already converted values through `pymavlink`, but every extractor should still record the raw message.

## Beginner Extraction Checklist

For every log:

1. Compute SHA256 before parsing.
2. Identify whether it is `.tlog` or `.bin`.
3. Count messages by type.
4. Extract first and last timestamps.
5. Extract vehicle/source IDs.
6. Extract GPS path.
7. Extract altitude fields.
8. Extract mode changes and events.
9. Extract battery/GPS quality.
10. Export normalized CSV/JSON and preserve raw messages.

## Red Flags in Logs

Potential evidence quality issues:

- No GPS lock.
- Sparse position messages.
- Sudden impossible position jumps.
- Telemetry link gaps in `.tlog`.
- `BAD_DATA` messages.
- Missing `SYSTEM_TIME`.
- Missing arming/disarming events.
- Very low satellite count.
- EKF variance or estimator warning.
- Data starts after takeoff or ends before landing.
